# AutoRAGEvals: A Self-Improving Loop for RAG Pipeline Optimization

AutoRAGEvals is an automated framework that iteratively optimizes a Retrieval-Augmented Generation (RAG) pipeline by using Claude as a meta-optimizer. It proposes hyperparameter changes one at a time, evaluates each with RAGAS and ROUGE-L, and keeps or reverts changes based on objective scoring — forming a closed optimization loop requiring no human intervention. The project also includes adversarial question generation, held-out generalization evaluation, and publication-quality visualizations of the optimization trajectory.

## Architecture

```
┌──────────────────────────────────────────────────────────────────────┐
│                        AutoRAGEvals Loop                             │
│                                                                      │
│   Document Corpus                                                    │
│   (HotpotQA dev)                                                     │
│        │                                                             │
│        ▼                                                             │
│   ┌─────────────┐    chunk_size, chunk_overlap,    ┌─────────────┐  │
│   │             │    top_k, prompt_variant,        │             │  │
│   │  RAG        │◄── reranker                      │   Claude    │  │
│   │  Pipeline   │                                  │  Optimizer  │  │
│   │  (LlamaIndex│                                  │             │  │
│   │  + ChromaDB)│                                  └──────┬──────┘  │
│   └──────┬──────┘                                         │         │
│          │                                         propose next     │
│          │ answers + contexts                      parameter change  │
│          ▼                                                 │         │
│   ┌─────────────┐    scores                        ┌──────┴──────┐  │
│   │   RAGAS     │──────────────────────────────────►   Keep /    │  │
│   │   Scorer    │  faithfulness, answer_relevancy,  │   Revert   │  │
│   │  + ROUGE-L  │  context_precision, context_recall│            │  │
│   └─────────────┘                                  └─────────────┘  │
│                                                                      │
│   Each config scored 3× → mean ± std reported                       │
│   Best config persisted across iterations (resumable)               │
└──────────────────────────────────────────────────────────────────────┘
```

## Project Structure

| File | Description |
|---|---|
| `pipeline.py` | Core `RAGPipeline` class — LlamaIndex + ChromaDB + OpenAI. Configurable chunking, retrieval, prompt variants, and optional reranker. |
| `load_data.py` | Downloads and parses HotpotQA dev set from HuggingFace; returns deduplicated passage list and QA pairs. |
| `scorer.py` | Wraps RAGAS `evaluate()` — runs faithfulness, answer_relevancy, context_precision, context_recall. Handles asyncio/event-loop conflicts with LlamaIndex. |
| `generate_questions.py` | Uses Claude to generate 50 adversarial questions (multi-hop, ambiguous, out-of-scope, paraphrase) from HotpotQA passages. |
| `optimizer.py` | Main optimization loop — asks Claude to propose config changes, scores each with RAGAS×3 runs (mean ± std), keeps or reverts, logs everything to JSON. Fully resumable. |
| `run_optimizer.py` | Entry point for the full 15-iteration optimization run. |
| `test_optimizer.py` | Smoke test — 2 iterations on 10 questions with a 100-passage corpus. Verifies the loop works end-to-end. |
| `smoke_test.py` | End-to-end pipeline test on 5 HotpotQA questions with RAGAS scoring. |
| `held_out_eval.py` | Evaluates baseline and best configs on `hotpot_heldout_50.json` — 50 questions never seen during optimization. |
| `visualizations.py` | Generates four publication-quality figures (see [Figures](#figures) below). |
| `case_studies.py` | Selects and formats 3 illustrative case studies from the held-out results: improved, stayed wrong, and ROUGE-L vs RAGAS disagreement. |
| `adversarial_questions.json` | 50 adversarial questions generated by Claude (38 scoreable + 12 out-of-scope). |
| `hotpot_heldout_50.json` | 50 randomly sampled HotpotQA questions held out from optimization (seed=42, yes/no filtered). |
| `experiment_log.json` | Full optimizer run log — every iteration's config, scores (mean ± std), proposal, and keep/revert decision. |
| `held_out_results.json` | Per-question answers, contexts, and aggregate scores for both configs on the held-out set. |
| `case_studies.json` | Structured output of the three case studies. |
| `requirements.txt` | Pinned Python dependencies. |
| `.env.example` | Template for required API keys. |
| `figures/` | Output directory for PNG charts. |

## Setup

### 1. Clone the repository

```bash
git clone https://github.com/randomdrth/AutoResearch-for-RAG.git
cd AutoResearch-for-RAG
```

### 2. Install dependencies

Python 3.9+ required.

```bash
pip install -r requirements.txt
```

### 3. Configure API keys

```bash
cp .env.example .env
```

Open `.env` and fill in:

```
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
```

Both keys are required. OpenAI is used for embeddings (`text-embedding-3-small`) and RAG generation/judging (`gpt-4o-mini`). Anthropic is used for adversarial question generation and the optimizer proposals (Claude Sonnet).

Optionally, add `COHERE_API_KEY` to use `CohereRerank` instead of the LLM-based reranker when `reranker=True` is proposed.

### 4. Download the HotpotQA dev set

Download `hotpot_dev_fullwiki_v1.json` from the [HotpotQA website](https://hotpotqa.github.io/) and place it in the project root.

## How to Run

Run in this order:

```bash
# 1. Verify the pipeline works end-to-end (5 questions, RAGAS scored)
python smoke_test.py

# 2. Generate adversarial evaluation questions via Claude
python generate_questions.py

# 3. Smoke test the optimizer loop (2 iterations, 10 questions, ~10 min)
python test_optimizer.py

# 4. Run the full optimization (15 iterations, 38 questions, 3 runs each, ~4–6 hrs)
python run_optimizer.py

# 5. Evaluate generalization on the held-out set
python held_out_eval.py

# 6. Generate figures
python visualizations.py

# 7. Generate case studies
python case_studies.py
```

`run_optimizer.py` is **resumable** — if interrupted, re-running it will pick up from the last completed iteration using `experiment_log.json`.

## Key Results

### Optimization trajectory (adversarial set, 38 questions)

| Iteration | Change | Overall RAGAS | Δ | Decision |
|---|---|---|---|---|
| 0 | Baseline (top_k=3) | 0.7019 | — | Kept |
| 1 | top_k: 3 → 5 | 0.7475 | +0.0456 | Kept |
| 2 | top_k: 5 → 7 | 0.7807 | +0.0332 | **Kept (best)** |
| 3 | chunk_overlap: 50 → 100 | 0.7581 | −0.0226 | Reverted |
| 4 | chunk_overlap: 50 → 25 | 0.7386 | −0.0421 | Reverted |
| 5 | chunk_size: 512 → 1024 | 0.7380 | −0.0427 | Reverted |
| 6 | prompt_variant: chain_of_thought | 0.7368 ± 0.013 | −0.0439 | Reverted |
| 7 | prompt_variant: concise | 0.7434 ± 0.011 | −0.0373 | Reverted |

### Baseline vs best config — all metrics

| Metric | Adversarial Baseline | Adversarial Best | Δ | Held-out Baseline | Held-out Best | Δ |
|---|---|---|---|---|---|---|
| Faithfulness | 0.8377 | 0.9057 | +0.068 | 0.3767 | 0.4691 | +0.092 |
| Answer Relevancy | 0.7683 | 0.8034 | +0.035 | 0.6380 | 0.6064 | −0.032 |
| Context Precision | 0.5614 | 0.6376 | +0.076 | 0.1600 | 0.1600 | +0.000 |
| Context Recall | 0.6404 | 0.7763 | +0.136 | 0.1200 | 0.1400 | +0.020 |
| **Overall** | **0.7019** | **0.7807** | **+0.079** | **0.3237** | **0.3439** | **+0.020** |
| ROUGE-L | 0.1449 | 0.1596 | +0.015 | 0.0901 | 0.0838 | −0.006 |

**Best config:** `chunk_size=512, chunk_overlap=50, top_k=7, prompt_variant=baseline, reranker=False`

## Key Findings

- **Retrieval breadth was the only lever that helped.** Increasing `top_k` from 3→5→7 drove the entire optimization gain (+7.9 pp overall). All chunking and prompt changes reverted. The default LlamaIndex prompt is already well-tuned for HotpotQA-style multi-hop questions.

- **There is a large generalization gap.** The best config scores 0.7807 on the adversarial set but only 0.3439 on the held-out HotpotQA set (−0.437). This gap is driven by retrieval coverage: only 300 of 7,405 HotpotQA passages are indexed, so supporting facts are frequently missing from the corpus entirely.

- **ROUGE-L and RAGAS can move in opposite directions.** The `concise` prompt variant (iter 7) nearly doubled ROUGE-L (0.160→0.204) while RAGAS overall fell (−0.037). Conversely, in the held-out case studies, the best config sometimes refused to answer when its 7 retrieved chunks were noisy — increasing RAGAS faithfulness while collapsing ROUGE-L to near zero. This demonstrates that the two metrics capture meaningfully different quality dimensions.

- **Statistical robustness matters even at this scale.** Iters 6–7 showed per-config standard deviations of ±0.011–0.013 on overall RAGAS across 3 runs — roughly the same magnitude as the keep/revert margins observed in later iterations. Single-run scoring would have produced unreliable decisions for these experiments.
